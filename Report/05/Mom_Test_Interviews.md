# Report/05 — Mom Test 인터뷰 (UnitConverter_04)

| 항목 | 내용 |
|------|------|
| 작성일 | 2025-06-05 |
| 방법 | Rob Fitzpatrick Mom Test 스타일 1:1 인터뷰 |
| 회차 | 3회 (서로 다른 페르소나) |
| 목적 | 길이 단위(meter/feet/yard) 변환 **실제 불편** 수집 → 주제 선정 근거 |

---

## 인터뷰 규칙

1. 한 번에 질문 1개만
2. **금지:** "만들면 좋겠어?", "이 기능 필요해?", "TDD 하면 어때?"
3. **허용:** "마지막으로 ~했을 때", "그때 몇 분 걸렸어?", "뭐 때문에 포기했어?"
4. 답이 모호하면 추궁 1개만
5. 솔루션 언급 시 "그건 나중" — 불편·과거 행동으로 되돌림

---

## 인터뷰 1 — 백엔드 개발자 (경력 4년)

**페르소나:** 물류 대시보드, 내부 DB meter / 화면 feet·yard

| # | 질문 (요약) | 답 (요약) |
|---|-------------|-----------|
| Q1 | 마지막으로 변환 코드 연 이유 | 3주 전, yard 화면 출력 요청 |
| Q2 | 숫자 맞는지 확인에 걸린 시간 | 예상 5분 → **40분+** (yard 0.01 어긋남) |
| Q3 | 0.01 차이 발견 경로 | QA 스테이징 슬랙, 로컬 반올림으로 미발견 |
| Q4 | 40분 중 가장 오래 걸린 것 | feet↔yard **meter 미경유** 직접 비율, `convert` 함수 추적 |
| Q5 | 이전에 변환 코드 짠 적 / 불편 | 8개월 전 30줄 if/elif, **비율 검증이 본인 머리+계산기** |
| Q6 | 동료 feet→yard 문의 빈도 | 2달 전 스프린트 **3회**, 회당 3~4분 |
| Q7 | yard 추가 시 포기/막힘 | 비율 **두 군데** 하드코딩 → "나중에 정리" 후 복붙 |
| Q8 | 정리 실제 수행 여부 | **없음** — 우선순위·건드리기 싫음 |
| Q9 | 3주 전 버그 때 제일 짜증 | 금요일 오후 QA 왕복, 원인 1줄인데 **2시간 30분** |
| Q10 | 사람 간 소통 시간 | 슬랙·설명·재현 **~30분** |
| Q11 | 새 단위(inch) 추가 | 5개월 전, 견적 반나절 → 실제 **1일**, feet 값 미세 변화 미인지 |
| Q12 | 손으로 확인한 입력 | `meter:2.5`, `feet:1`, `meter:1`(yard만) — **음수·형식 미확인** |
| Q13 | 사용자 입력 터진 마지막 건 | `meter:-1` — 이상 출력 → CS 티켓 |
| Q14 | CS 티켓 처리 시간 | **40분** |
| Q15 | 돈/일정으로 아픈 순간 | 금요일 "작은 화면 수정" → **2시간 30분** |
| Q16 | 다시 겪고 싶지 않은 것 | 비율 여러 곳 + **검증이 본인 머리·계산기** |

## 인터뷰 2 — 데이터 엔지니어 (경력 6년)

**페르소나:** 센서·설비 ETL, CSV feet/meter 혼재, Airflow 배치

| # | 질문 (요약) | 답 (요약) |
|---|-------------|-----------|
| Q1 | 마지막으로 변환 코드 연 이유 | 6주 전, feet CSV → meter 스키마 + yard 컬럼 |
| Q2 | 첫 배치 통과까지 | 로컬 20분 → 스테이징 실패 **1시간 50분** |
| Q3 | null 3,200건 발견 경로 | 새벽 2시 Airflow 알람, yard 컬럼 전부 null |
| Q4 | 스킵 결정까지 | 원인 40분 + 슬랙 합의 **55분** |
| Q5 | yard null 직접 원인 | 헤더 `Yard` vs 코드 `yard` **대소문자** |
| Q6 | 변환 로직 위치 | `utils/convert.py` + **`legacy/fix_units.py` 중복** |
| Q7 | legacy 경로 모른 이유 | 문서 없음, 4주 전 추적 시 **비율 상이** 발견 |
| Q8 | 비율 차이 예 | `0.3048` vs `0.304` — 10ft 시 0.0008m, **한 달 뒤** 월합계 불일치 |
| Q9 | 월합계 불일치 대응 | SQL 비교 후 legacy 열기, **3시간** |
| Q10 | 음수 입력 마지막 사례 | 2개월 전 `feet:-0.5` 그대로 저장 → 이상치 알람 |
| Q11 | 음수 건 처리 시간 | 재적재·승인 **1시간 20분** (코드 수정 15분) |
| Q12 | 다시 겪고 싶지 않은 것 | **같은 변환·다른 비율** 2파일 → 한 달 뒤 3시간 추적 |

## 3회 공통 관찰 (사실만)

| 패턴 | 인터뷰 근거 |
|------|-------------|
| "작은 수정"이 40분~3시간으로 확대 | 1: 금요일 2.5h / 2: 온콜 55분+스킵 / 3: 형식 50분+yard 1h10m |
| 비율·로직이 여러 곳에 분산 | 1: elif 2곳 / 2: convert.py + legacy / 3: print·elif 여러 곳 |
| feet↔yard meter 미경유 시 어긋남 | 1: 0.01 QA / 3: 0.02 동료 |
| 검증이 수동·부분적 | 1: meter:2.5만 / 3: meter:1만 / 음수·형식은 사후 발견 |
| 단위 추가 시 시간·불안 | 1: inch 1일 / 3: cubit 2h+무테스트 |
| 음수·형식 오류는 뒤늦은 비용 | 1: meter:-1 CS 40분 / 2: feet:-0.5 재적재 1h20m |

## 인터뷰에서 나오지 않은 것 (주제 후보 제외)

- JSON/YAML 설정, csv/json 출력 포맷 — **과거 행동으로 미언급**
- TDD·ECB·registry 등 **솔루션 의향** — 피인터뷰이가 제안하지 않음
